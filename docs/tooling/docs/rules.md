# Documentation Rules

This document defines the baseline rules for maintaining project documentation.

## Rules

- Keep docs aligned with implemented behavior and public contracts
- Prefer short paragraphs and concrete wording
- Update documentation in the same change stream as the behavior change
- Use placeholders instead of invented detail when information is unknown
- Remove or simplify documents that no longer improve clarity

## Style guidelines

- Use clear, simple language
- Prefer active voice
- Be specific rather than vague
- Include examples where helpful

## Review trigger

Revisit documentation whenever:
- Architecture changes
- Interfaces change
- Tooling changes
- Workflows change
